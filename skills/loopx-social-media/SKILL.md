# LoopX Social Media

Use this skill to produce public-safe LoopX creator/social output without
turning LoopX into an ungated publisher. The default target is X, but the same
source-map and gate discipline can support other public channels.

## Workflow

1. Define the launch objective: repo visit, star/watch, issue discussion,
   pilot lead, buyer conversation, or technical narrative credibility.
2. Build a source map before drafting. Separate public repo/docs, public social
   research, owner-approved CS-Notes material, generated visuals, and gated
   private context.
3. Pick one angle family:
   - fastest local entry into loop engineering;
   - philosophy of loop engineering and human judgment;
   - academic framing such as handoff fixed points or state drift;
   - product proof from gates, quota, evidence, handoff, and safe fallback;
   - connector/value story such as GitHub reply monitoring or X via ego-lite.
4. Draft for one concrete reader. Prefer AI tool power users, open-source
   maintainers, developer-tool builders, founders, or research engineers.
5. Add a visual plan. X launch posts should usually include one strong image:
   a simple diagram, polished repo/product screenshot, or generated launch
   card. Use two images only when the second adds proof, not decoration.
6. Add the repo link when the post is about LoopX itself:
   `https://github.com/huangruiteng/loopx`.
7. Use mentions sparingly. A first launch post may mention one or two highly
   relevant people; do not mass-tag or tag people unrelated to the claim.
8. Select timing. Read `references/x-launch-playbook.md` for the current X
   timing heuristic, then schedule into the next suitable local window.
9. Produce a no-send packet with body, image paths, mentions, source map,
   timing, anti-spam checks, and the exact publish gate.
10. Do not publish, reply, edit, delete, appeal, or bump without explicit user
    approval for the exact external action.

## X Positioning Rules

- Say `loop engineering` when that is the market/category claim.
- Avoid making "control plane" sound like a dashboard. Define it as the layer
  around the loop: state, heartbeat, gates, evidence, quota, and handoff.
- Emphasize local-first ease of adoption: "fastest local entry" and "works
  around Codex, Claude Code, Cursor, and other agent runtimes" are stronger
  than abstract architecture.
- Keep the human role clear: humans keep judgment; agents keep safe lanes
  moving.
- Prefer concrete verbs: connect, resume, gate, validate, hand off, monitor.
- Avoid inflated claims such as autonomous production action, guaranteed
  virality, benchmark uplift without paired evidence, or private customer
  proof.

## Boundary Rules

- Public repo files and public docs may be summarized and linked.
- CS-Notes material may inform mechanisms, angles, and image/storyboard
  choices only when transformed into public-safe LoopX language.
- Do not copy private source bodies, internal/company context, local paths,
  credentials, raw chat logs, raw benchmark traces, or unapproved screenshots.
- Browser/X/ego-lite use must stay terms-aware: read public material only as
  needed, avoid timeline dumps, and treat publishing as an explicit gate.
- If a GitHub or X account is cold/new, reduce spam shape: unique first
  paragraph, target-specific evidence, concrete ask, and clear stop condition.
